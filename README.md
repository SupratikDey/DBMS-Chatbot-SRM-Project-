MongoDB's flexible schema allows us to change the structure of our documents without performing complex migrations.MongoDB, a NoSQL database, was selected for this project for several reasons:### Why MongoDB was Chosen    -   **Our Implementation:** We avoid transitive dependencies by referencing documents in other collections. For example, instead of storing user details directly in the `ChatHistory`, we store a `userId` which references the `Users` collection. This means if a `username` changes, we only have to update it in one place (the `Users` collection), not in every chat history record.    -   **Rule:** The table must be in 2NF, and there should be no transitive dependencies (where a non-key attribute depends on another non-key attribute).-   **Third Normal Form (3NF):**    -   **Our Implementation:** Our collections use a single `_id` field as the primary key, so there are no partial dependencies. For instance, in the `Questions` collection, `text` and `subject` are dependent on the `_id` of the question, not just a part of it.    -   **Rule:** The table must be in 1NF, and all non-key attributes must be fully functionally dependent on the primary key. This is mainly for composite primary keys.-   **Second Normal Form (2NF):**    -   **Our Implementation:** All our schemas adhere to this. For example, in the `Users` schema, each document has a unique `_id` and fields like `username` hold atomic values. The `relatedQuestions` in `Concepts` is an array of `ObjectId`s, where each `ObjectId` is a single, atomic value.    -   **Rule:** Ensures that table cells hold a single value and that records are unique.-   **First Normal Form (1NF):**Even though MongoDB is schema-less, the principles of normalization are relevant for designing clean and efficient data models.### Normalization Explanation    - A `Question` can be related to many `Concepts`, and a `Concept` can relate to many `Questions` (Many-to-Many, implemented via an array of references in the `Concepts` entity).    - A `Question` is part of one `ChatHistory` record (One-to-One, from ChatHistory's perspective).    - A `User` can have many `ChatHistory` records (One-to-Many).- **Relationships:**        - Attributes: `_id` (Primary Key), `name`, `description`, `subject`, `relatedQuestions` (Array of Foreign Keys to Questions)    - `Concepts`: Stores academic concepts related to questions.        - Attributes: `_id` (Primary Key), `userId` (Foreign Key to Users), `questionId` (Foreign Key to Questions), `response`, `timestamp`    - `ChatHistory`: Logs the conversation between a user and the chatbot.        - Attributes: `_id` (Primary Key), `text`, `subject`, `createdAt`    - `Questions`: Stores the questions asked by users.        - Attributes: `_id` (Primary Key), `username`, `createdAt`    - `Users`: Stores user information.- **Entities:**Since we are using a NoSQL database (MongoDB), we don't have a traditional ER diagram with physical table links.
